# Pre-Blocking Verification Checklist

실제 차단 기능을 연결하기 전에 반드시 확인해야 하는 검증 목록입니다.

목표는 명확합니다.

- 사용자가 기기 제어권을 잃지 않는다.
- Safe Mode, Kill Switch, Emergency Unlock이 항상 우선한다.
- 정책 저장/계산/감지는 정상 동작한다.
- AccessibilityService는 아직 실제 차단을 실행하지 않는다.

## 0. 빌드와 설치

검증 명령:

```bat
set "JAVA_HOME=C:\Program Files\Android\Android Studio2\jbr" && set "PATH=%JAVA_HOME%\bin;%PATH%" && .\gradlew.bat installDebug -Pandroid.injected.device.serial=emulator-5554
```

기대 결과:

- `BUILD SUCCESSFUL`
- 앱이 대상 기기 또는 에뮬레이터에 설치됨
- 앱 실행 시 크래시 없음

## 1. 기본 실행

절차:

1. 앱 실행
2. 첫 화면이 표시되는지 확인
3. 하단 또는 상단 탭이 정상 표시되는지 확인
4. 앱을 백그라운드로 보냈다가 다시 열기

기대 결과:

- 앱이 멈추거나 몇 초 이상 얼지 않음
- 탭 전환이 가능함
- 기본 언어는 한국어
- Safe Mode는 기본 ON

## 2. 탭 이동과 UI

검증 대상:

- 개요
- 요일/그룹
- 앱
- 안전
- 설정

절차:

1. 각 탭을 터치해서 이동
2. 좌우 스와이프로 탭 이동
3. 작은 화면에서 탭 이름이 잘리지 않는지 확인
4. 오른쪽/왼쪽에 추가 탭이 있을 때 힌트가 보이는지 확인
5. 화면 회전 또는 태블릿 화면에서 레이아웃 확인

기대 결과:

- 선택된 탭이 명확히 표시됨
- 탭 이름이 과도하게 잘리지 않음
- 스와이프 전환이 자연스러움
- 카드/버튼/텍스트가 겹치지 않음

## 3. Safe Mode

절차:

1. 안전 탭으로 이동
2. Safe Mode ON 확인
3. Safe Mode OFF로 변경
4. 다시 ON으로 변경
5. 앱 재실행 후 상태 유지 확인

기대 결과:

- ON/OFF 상태가 즉시 UI에 반영됨
- 앱 재실행 후에도 저장된 상태 유지
- Safe Mode ON 상태에서는 차단 시뮬레이션 결과가 Safe Mode로 표시됨
- 실제 차단은 발생하지 않음

## 4. Kill Switch

절차:

1. Safe Mode OFF
2. Policy Enforcement ON
3. 안전 탭에서 Kill Switch 실행

기대 결과:

- Safe Mode가 즉시 ON으로 변경됨
- Policy Enforcement가 비활성화됨
- 이벤트 로그에 Kill Switch 기록이 남음
- 앱이나 기기가 잠기지 않음

## 5. Emergency Unlock

절차:

1. 안전 탭의 긴급 해제 PIN 입력
2. 잘못된 PIN 입력
3. 올바른 PIN 입력

기대 결과:

- 잘못된 PIN은 실패로 표시됨
- 올바른 PIN은 Safe Mode를 ON으로 전환
- 인터넷 연결 없이 동작
- 앱 재실행 후 Safe Mode ON 유지

## 6. PIN 설정

절차:

1. 설정 탭으로 이동
2. 관리 PIN 변경
3. 긴급 PIN 변경
4. 잘못된 현재 PIN으로 변경 시도
5. 새 PIN으로 정책 저장 또는 긴급 해제 검증

기대 결과:

- 현재 PIN이 틀리면 변경 실패
- 새 PIN은 4자리 이상일 때만 유효
- 관리 PIN과 긴급 PIN이 각각 독립적으로 동작
- 이벤트 또는 상태 메시지가 정상 표시됨

## 7. UsageStats 권한

절차:

1. 개요 탭으로 이동
2. 사용정보 권한 상태 확인
3. 권한이 없으면 권한 설정 버튼 클릭
4. Android 설정에서 Screen Time Manager 사용정보 접근 허용
5. 앱으로 돌아와 새로고침

기대 결과:

- 권한이 없을 때 안내가 표시됨
- 권한 허용 후 오늘 사용 목록이 표시됨
- 앱 이름과 아이콘이 표시됨
- 패키지명만 보이는 앱은 최소화되어야 함

## 8. 오늘 사용 시간

절차:

1. Chrome, YouTube, 카카오톡 등 실제 앱을 20초 이상 사용
2. Screen Time Manager로 돌아오기
3. 개요 탭에서 새로고침

기대 결과:

- 실제로 화면에 띄워 사용한 앱만 집계됨
- 많이 사용한 앱이 위에 표시됨
- 백그라운드 앱은 사용 시간에 포함되지 않음
- 시스템 UI, 키보드, 런처, 권한 컨트롤러는 표시되지 않음
- 10초 미만 사용 앱은 표시되지 않을 수 있음

## 9. 오늘 기준 시간대

절차:

1. 자정 전후 또는 날짜 변경 후 사용량 확인
2. 한국 시간 기준으로 오늘 사용량이 리셋되는지 확인

기대 결과:

- 현재 기기 로컬 시간 기준 오늘 범위로 집계됨
- 한국 시간으로 설정된 기기에서는 한국 날짜 기준으로 표시됨

## 10. 정책 요약

절차:

1. 전체 일일 제한을 낮게 설정
2. 앱별 제한 또는 그룹 제한 설정
3. 개요 탭의 정책 요약 확인

기대 결과:

- 전체 사용량 / 전체 제한 표시
- 그룹 사용량 / 그룹 제한 표시
- 앱별 사용량 / 앱별 제한 표시
- 정상은 초록, 근접은 주황, 초과는 빨강 계열로 표현
- Edit 버튼을 누르면 요일/그룹 탭으로 이동

## 11. 요일별 제한

절차:

1. 요일/그룹 탭으로 이동
2. 월~일 각각 다른 제한 시간 설정
3. 평일 버튼 적용
4. 주말 버튼 적용
5. 관리 PIN 입력 후 저장
6. 앱 재실행 후 값 유지 확인

기대 결과:

- 선택한 요일 값이 즉시 UI에 반영됨
- 5분 또는 15분 단위 조절이 자연스러움
- 저장 전 변경 상태가 표시됨
- 저장 후 최신 상태로 표시됨
- 오늘 요일의 제한이 개요 정책 요약에 반영됨

## 12. 앱 그룹

절차:

1. 요일/그룹 탭으로 이동
2. 새 그룹 추가
3. 그룹 이름 입력
4. 그룹 예산 설정
5. 앱을 그룹에 추가
6. 저장 후 앱 재실행

기대 결과:

- 그룹 추가/수정/삭제 가능
- 앱은 그룹에 포함됨으로 표시됨
- 같은 앱이 여러 그룹에 중복 포함되지 않음
- 그룹 예산이 정책 요약과 차단 시뮬레이션에 반영됨
- 시스템 앱과 키보드 앱은 그룹 선택 목록에 보이지 않음

## 13. 앱별 제한

절차:

1. 앱 탭으로 이동
2. 검색으로 앱 찾기
3. 앱 항목 선택
4. 바텀 시트에서 제한 없음, 5분, 10분, 15분, 30분 선택
5. 슬라이더와 -5/+5 버튼으로 미세 조정
6. 적용
7. 관리 PIN 입력 후 저장

기대 결과:

- 바텀 시트가 충분히 위로 올라와 버튼이 가려지지 않음
- 제한 없음/제한 상태가 리스트에 바로 반영됨
- 앱별 제한 값이 저장 후 유지됨
- 제한 앱 필터와 미제한 앱 필터가 정상 동작
- 시스템 앱과 키보드 앱은 앱 목록에 보이지 않음

## 14. 정책 예산 검증

절차:

1. 특정 요일 전체 제한을 60분으로 설정
2. 앱별 제한 합계를 60분 초과로 설정
3. 저장 시도
4. 그룹 예산 합계도 전체 제한 초과로 설정 후 저장 시도

기대 결과:

- 앱별 제한 총합이 해당 요일 전체 제한을 넘으면 저장 차단
- 그룹 예산 총합이 해당 요일 전체 제한을 넘으면 저장 차단
- 초과 요일이 UI에 표시됨
- 잘못된 정책이 저장되지 않음

## 15. 언어 설정

절차:

1. 설정 탭으로 이동
2. 한국어/영어 변경
3. 각 탭 문구 확인
4. 앱 재실행 후 언어 유지 확인

기대 결과:

- 기본값은 한국어
- 선택한 언어가 즉시 반영됨
- 앱 재실행 후에도 유지됨
- 주요 화면에 깨진 문자열이 없음

## 16. Android 알림

절차:

1. 설정 또는 안전 탭에서 알림 권한 상태 확인
2. Android 13 이상이면 알림 권한 허용
3. 제한 80% 근접 또는 초과 상태를 만들기
4. 일정 시간 후 이벤트 로그와 알림 확인

기대 결과:

- 권한 없으면 알림 요청 가능
- 권한 허용 후 정책 경고/초과 알림 가능
- 같은 대상의 같은 상태 알림이 과도하게 반복되지 않음
- Safe Mode ON 또는 Policy Enforcement OFF에서는 알림/로그가 정책 적용처럼 동작하지 않음

## 17. WorkManager 백그라운드 체크

절차:

1. Safe Mode OFF
2. Policy Enforcement ON
3. UsageStats 권한 ON
4. 제한 초과 상태 만들기
5. 앱을 백그라운드로 보낸 뒤 일정 시간 대기
6. 이벤트 로그 확인

기대 결과:

- 제한 상태가 이벤트 로그에 기록될 수 있음
- 알림 권한이 있으면 알림이 표시될 수 있음
- 실제 차단은 발생하지 않음
- Safe Mode ON에서는 백그라운드 정책 체크가 강제 동작하지 않음

## 18. AccessibilityService 권한

절차:

1. 안전 탭으로 이동
2. 접근성 설정 버튼 클릭
3. Android 접근성 설정에서 Screen Time Manager 활성화
4. 앱으로 돌아와 준비 상태 확인

기대 결과:

- 접근성 권한 상태가 준비 상태에 반영됨
- 서비스가 켜져도 실제 차단은 발생하지 않음
- 이벤트 로그에 서비스 연결 기록이 남을 수 있음

## 19. 최근 감지

절차:

1. Safe Mode OFF
2. Policy Enforcement ON
3. AccessibilityService ON
4. Chrome 같은 일반 앱 실행
5. Screen Time Manager로 돌아와 안전 탭 확인
6. 키보드를 띄운 뒤 다시 안전 탭 확인
7. Android Settings 앱 실행 후 다시 안전 탭 확인

기대 결과:

- 일반 사용자 앱은 최근 감지에 표시될 수 있음
- 키보드 앱은 최근 감지에 표시되지 않음
- Settings, System UI, 런처, 권한 컨트롤러는 최근 감지에 표시되지 않음
- whitelist 앱은 차단 평가 대상이 아님

## 20. 차단 시뮬레이션

절차:

1. Safe Mode ON 상태에서 확인
2. Safe Mode OFF, Policy Enforcement OFF 상태에서 확인
3. Safe Mode OFF, Policy Enforcement ON 상태에서 확인
4. 앱별 제한 초과 상태 만들기
5. 그룹 제한 초과 상태 만들기
6. 전체 제한 초과 상태 만들기

기대 결과:

- Safe Mode ON이면 결과가 Safe Mode로 표시됨
- Policy Enforcement OFF이면 정책 OFF로 표시됨
- 제한이 없으면 제한 없음 또는 허용으로 표시됨
- 제한 초과 시 would-block 성격의 결과가 표시됨
- 실제 차단은 발생하지 않음

## 21. 차단 화면 미리보기

절차:

1. would-block 대상 앱이 생기도록 제한 설정
2. 안전 탭에서 차단 화면 미리보기 버튼 확인
3. 버튼 클릭
4. 미리보기 화면에서 닫기
5. 다시 열어서 Kill Switch 실행
6. 다시 열어서 긴급 PIN 입력

기대 결과:

- 차단 화면 형태가 표시됨
- 미리보기 배지가 표시됨
- 닫기 또는 뒤로 가기가 가능
- Kill Switch 실행 시 Safe Mode ON으로 복구
- 긴급 PIN 성공 시 화면이 닫히고 Safe Mode ON
- 실제 앱 실행 중 자동으로 이 화면이 뜨지는 않음

## 22. 실제 차단 미연결 확인

절차:

1. Safe Mode OFF
2. Policy Enforcement ON
3. AccessibilityService ON
4. Chrome 제한을 초과 상태로 만들기
5. Chrome 실행

기대 결과:

- Chrome이 자동으로 막히지 않음
- Screen Time Manager 차단 화면이 자동으로 뜨지 않음
- 이벤트 로그에는 `would be blocked` 성격의 기록만 남을 수 있음

이 항목이 실패하면 실제 차단 단계로 넘어가면 안 됩니다.

## 23. whitelist 필수 앱

절대 차단/감지 UI 노출 제외 대상:

- Android Settings
- Google Play Store
- Package Installer
- Permission Controller
- System UI
- Launcher / One UI Home
- Keyboard / Input Method
- Dialer / Phone
- Messaging
- Screen Time Manager

검증 기대 결과:

- 위 앱들은 앱별 제한 목록에 보이지 않음
- 위 앱들은 최근 감지에 보이지 않음
- 위 앱들은 would-block 대상으로 표시되지 않음
- 실제 차단 단계에서도 반드시 제외되어야 함

## 24. 이벤트 로그

절차:

1. Safe Mode 변경
2. Kill Switch 실행
3. 정책 저장
4. AccessibilityService ON/OFF
5. 설정 탭에서 이벤트 로그 확인
6. 이벤트 로그 지우기

기대 결과:

- 주요 안전 이벤트가 기록됨
- 이벤트 로그 삭제 가능
- 삭제 후 중복 알림 방지 키도 초기화됨

## 25. 크래시/복구

절차:

1. 앱을 강제 종료
2. 다시 실행
3. Safe Mode 상태 확인

기대 결과:

- 비정상 종료로 판단되면 Safe Mode ON 복구
- 앱 실행 불능 상태가 되지 않음
- 사용자가 계속 앱을 열고 제어할 수 있음

## 26. 성능

절차:

1. 앱 실행 직후 탭 전환
2. 백그라운드에서 복귀 후 탭 전환
3. 앱 목록이 많은 기기에서 앱 탭 스크롤
4. 검색어 입력
5. 바텀 시트 열기/닫기

기대 결과:

- 탭 전환이 눈에 띄게 끊기지 않음
- 백그라운드 복귀 후 긴 멈춤 없음
- 앱 목록 스크롤이 안정적
- 키보드 입력 시 최근 감지에 키보드가 뜨지 않음

## 27. 최종 승인 기준

실제 차단 구현으로 넘어가기 위한 최소 기준:

- Safe Mode ON/OFF 정상
- Kill Switch 정상
- Emergency Unlock 정상
- UsageStats 집계 정상
- 시스템/키보드/런처 앱 필터 정상
- 요일별 제한 저장 정상
- 앱 그룹 저장 정상
- 앱별 제한 저장 정상
